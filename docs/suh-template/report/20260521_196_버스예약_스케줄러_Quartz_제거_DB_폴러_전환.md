# 버스예약 자동 예약 스케줄러 Quartz 제거 및 DB 폴러 전환

## 개요

기존 Quartz in-memory JobStore 기반 자동 예약 스케줄러는 서버 재배포 시 Trigger 가 휘발되고, 초기 Trigger 가 항상 "내일 22~23시" 로 설정되어 기동 첫날 발화하지 못하는 구조적 결함이 있었다. 이를 해결하기 위해 Quartz 의존성·코드를 완전히 제거하고 DB 단일 row 테이블에 `nextFireAt` 을 영속화하는 Spring `@Scheduled` 폴러로 전환했다. 재배포 무관 동작, 누락 발화 복구(당일 자정 컷오프), 다음 허용 요일 윈도우 계산 등 안정성 요구사항을 모두 반영했다.

## 주요 구현 내용

### DB 영속 nextFireAt 폴러 패턴

`somansa_bus_scheduler_state` 테이블의 단일 row 에 다음 발화 예정 시각을 저장한다. `SomansaBusSchedulerPoller` 가 10분마다 `tick()` 을 호출하며, `tick()` 은 영속된 `nextFireAt` 과 현재 시각을 비교해 발화 여부를 결정한다. 재배포 후 부팅 시 첫 10분 폴링 내에 영속 값을 그대로 읽어들이므로 Trigger 휘발 문제가 원천 차단된다.

### 트랜잭션 격리 전략

`tick()` 자체는 비트랜잭션으로 두고, `loadOrInitState`, `updateNextFireAtOnly`, `markStateAfterFire` 등 DB 갱신 메서드만 개별 `@Transactional` 적용한다. 자동 예약 호출(`reservationService.scheduledAutoReservation()`) 은 자체 트랜잭션을 보유하며, 호출부에서 try-catch 로 예외를 삼킨다. 이로써 reservation 실패가 state UPDATE 를 롤백시켜 `nextFireAt` 무갱신 → 다음 폴링마다 동일 시각 무한 재시도 패턴을 방지한다.

### 당일 자정 컷오프

발화 시각이 지난 후 폴링이 도래해도, `nextFireAt` 의 같은 날짜 안에 폴링한 경우에만 실제 예약을 시도한다. 다음날을 넘긴 누락은 이미 의미가 없으므로 (당일 예약 윈도우 종료) `nextFireAt` 만 갱신하고 발화는 skip 한다. 22~23시 윈도우 직후 서버가 꺼졌다가 다음날 오전에 재기동되는 시나리오에서 무의미한 지연 발화를 차단한다.

### 다음 허용 요일 윈도우 계산

`computeNextFireAt(referenceTime)` 은 ServerOption 의 `SOMANSA_BUS_SCHEDULER_DAYS` 설정을 반영하여 다음 발화 후보 날짜가 비허용 요일이면 허용 요일까지 전진시킨다. 무한 루프 방지를 위해 8회 가드를 두어, 허용 요일 설정이 비어있는 경우 다음날을 fallback 으로 사용한다. fromHour~toHour 범위 내 random 분 값으로 정확한 발화 시각을 결정한다.

### 싱글톤 row 패턴

`SINGLETON_ID = UUID.fromString("00000000-0000-0000-0000-000000000001")` 상수를 통해 항상 동일한 row 만 조회/저장한다. `@GeneratedValue` 미적용 + 수동 UUID 지정. 다른 row 가 수동 INSERT 되더라도 `findById(SINGLETON_ID)` 사용으로 격리된다.

### 예외 처리 매트릭스

- ServerOption 키 미존재 → `getOptionValue` 가 enum defaultValue 반환
- `TIME_FROM`/`TIME_TO` 비숫자 → `NumberFormatException` catch → 기본값(22/23) 사용 + warn 로그
- `DAYS` 파싱 실패/빈값 → maxIter 가드로 fallback (다음날 사용) + error 로그
- reservation 호출 예외 → 내부 catch, `lastFiredAt` 미갱신, `nextFireAt` 정상 갱신
- DB 연결 끊김 → `tick()` 예외 → poller catch → 다음 폴링 재시도
- 폴러 자체 예외 → 다음 fixedDelay 영향 없음 (Spring TaskScheduler 보장)

## 주의사항

- `daysAhead` 는 현 상태(`int daysAhead = 1` 하드코딩) 그대로 유지. 정원초과로 실패할 가능성이 있으나 별도 이슈로 분리
- in-memory Quartz 와 마찬가지로 단일 인스턴스 가정. 멀티 인스턴스 배포 시 분산 락 필요 (현재 scope 아님)
- 폴링 주기 10분 — 발화 윈도우(22~23시, 2시간)보다 충분히 짧아 누락 0 보장. 단 컷오프 초과 케이스에서 최대 10분 지연 가능
- 마이그레이션 `V2_5_35` 은 `CREATE TABLE IF NOT EXISTS` 로 작성되어 초기 DB / 기존 DB 모두 안전. Quartz `qrtz_*` 테이블은 JDBC JobStore 미사용으로 애초에 존재하지 않아 DROP 불필요
- 통합 테스트는 dev 프로필 DB 직접 사용, mock 없음. 활성 SomansaBusSchedule 이 비어있는 dev 환경 가정으로 외부 API 호출 0회 보장
