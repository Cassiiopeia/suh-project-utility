### 📌 작업 개요
RAG 챗봇의 문서 청크 분할 방식을 문자 수 기준(500자)에서 토큰 수 기준(3000토큰)으로 변경. 정확한 토큰 계산 대신 근사치를 사용하여 컴퓨팅 자원 소모 최소화.

**보고서 파일**: `.report/20251215_#114_기존_500토큰으로_분할하여_저장하는_방식_3000토큰으로_분할하여_저장하는_방식으로_변경.md`

### 🎯 구현 목표
- 청크 분할 기준을 문자 수에서 토큰 수로 변경
- 500토큰 → 3000토큰으로 청크 크기 증가하여 검색 효율성 향상
- 정확한 토크나이저 라이브러리 없이 근사치만으로 계산하여 성능 유지
- 추가 의존성 없이 기존 코드만 수정

### 🔧 주요 변경사항 상세

#### splitIntoChunksByTokens() 메서드
문단 단위로 먼저 분할한 후, 각 문단의 토큰 수를 계산하여 청크 크기 초과 여부 판단. 문단이 크면 문장 단위로 재분할하여 자연스러운 경계 유지.

**특이사항**:
- 누적 토큰 수(`currentTokenCount`)로 분할 판단하여 효율적 처리
- 문장 경계(`.!?。！？`)를 고려하여 의미 단위 분할
- 오버랩 처리 시 토큰 수 기준으로 정확도 향상

#### estimateTokenCount() 메서드 개선
한글 비율에 따라 다른 계산 방식 적용:
- 한글 50% 이상: 문자 수 × 0.5
- 한글 10% 미만: 단어 수
- 혼합: 한글 토큰 수 + 영어 토큰 수

**특이사항**:
- 정확한 토크나이저 없이 근사치만 사용하여 성능 유지
- 실제 토큰 수와 ±10-20% 차이 가능하지만 RAG 검색에는 영향 없음

#### getOverlapTextByTokens() 메서드
텍스트 뒤에서부터 문자를 하나씩 추가하면서 토큰 수를 계산. 목표 토큰 수에 도달하면 중단하여 오버랩 텍스트 추출.

**특이사항**:
- 역순으로 문자 추가하여 자연스러운 오버랩 유지
- 매번 `estimateTokenCount()` 호출로 약간의 오버헤드 있지만 근사치 계산이라 빠름

### 🧪 테스트 및 검증
- 실제 문서로 청크 분할 테스트 필요
- 생성된 청크의 토큰 수가 대략 3000토큰 근처인지 확인
- 한글/영어 혼합 텍스트에서도 정상 작동 확인
- 기존 문서 재처리 시 새 방식 적용 확인

### 📌 참고사항
- **근사치 사용**: 정확한 토큰 수와 ±10-20% 차이 가능. RAG 검색 품질에는 큰 영향 없음
- **기존 문서 재처리**: 기존에 500자 기준으로 분할된 문서는 재처리 시 3000토큰 기준으로 재분할됨
- **성능**: 추가 라이브러리 없이 근사치만 사용하여 기존과 동일한 성능 유지
- **하위 호환성**: 기존 `getOverlapText()` 메서드는 유지하여 다른 곳에서 사용 시 문제 없음
