# 코드 점검 02 — 외부 연동 오류 처리

> 시리즈: [01 전체 요약](20260611_코드점검_01_전체요약.md) · **02 외부연동 오류처리** · 03 고정값→설정값(예정)

외부 서비스(소만사·깃허브·번역·AI서버·도커)를 호출할 때 **오류가 조용히 묻히는** 곳을 정리했습니다. 이번 버스 예약 문제처럼 "분명 켜놨는데 왜 안 되지?"가 재발하는 핵심 원인입니다.

---

## 공통 문제 패턴

1. **실패를 `-1` / `false` / `null`로만 표현** → 호출한 쪽은 "왜" 실패했는지 모름
2. **응답 본문이 비었을 때(NPE) 그대로 터짐** → `response.body().string()`에서 빈 응답 시 예외
3. **HTTP 오류(4xx/5xx)를 구분 안 함** → 404든 500이든 똑같이 "실패"로 뭉뚱그림
4. **리소스 누수** → `Response`를 try-with-resources 없이 사용해 연결이 안 닫힘

→ 공통 해법: **실패 시 `CustomException`을 던져** 무엇이 왜 실패했는지 로그·화면에 드러나게 하고, 응답 본문 null 체크와 try-with-resources를 일관 적용.

---

## 모듈별 상세

### 소만사 버스 (Suh-Domain-Somansa-Bus)

`SomansaBusApiService.java`
- `login()` — 로그인 페이지 GET 실패 / POST 실패 / 예외 모두 `-1` 반환. 실패 사유 손실
- `createSession()` — IOException 시 로그만 남기고 `false` 반환
- `makeReservation()` — 라인 187·192 `response.body() != null ? ... : "응답 없음"`, IOException 시 `false`만 반환

**개선:** 로그인/세션/예약 실패를 단계별 `CustomException`(예: 로그인 실패, 세션 실패, 예약 거부)으로 구분해 던진다.

추가: `SomansaBusHttpClient` — 현재 SSL 검증은 정상(우회 없음)이나 **연결/읽기 타임아웃 미설정**. 응답 없을 때 무한 대기 위험 → `connectTimeout`/`readTimeout`/`writeTimeout` 추가 권장.

### 깃허브 (Suh-Domain-Github)

`GithubService.java`
- 라인 114~127 — `Response`를 try-with-resources 없이 사용(리소스 누수), `response.body()` null 가능성 미처리

`IssueHelperService.java`
- 외부 호출부 일부에서 응답 검증 부실

**개선:** try-with-resources + body null 체크 + 상태코드별(404 vs 5xx) 예외 분기.

### 깃허브 잔디 (Suh-Domain-GrassPlanter)

`GrassService.java`
- 라인 366~403 `performGitHubCommit()` — GET 실패 시 아무 처리 없이 `sha=null`로 진행, 본문 null 미처리
- 라인 275~301 기여도 조회 — `response.body().string()` 리소스 누수, `Integer.parseInt(level)` NumberFormatException 미처리
- 라인 411~431 `extractSha()` — **JSON을 문자열로 직접 잘라 파싱**(코드 주석에도 "원래 Jackson 써야 함" 명시). 형식 바뀌면 깨짐

**개선:** 표준 JSON 파싱으로 교체 + 외부 호출 예외/리소스/파싱 안전화. (이 모듈은 곧 대개편 예정이므로 그때 함께 처리 가능)

### 번역 (Suh-Module-Translate)

`TranslateService.java`
- 라인 41·92 — `RuntimeException`으로 "텍스트 비었음" / "번역 오류" 던짐 → 사용자에게 500 에러

**개선:** `CustomException`(400: 입력없음 / 500: 번역실패)으로 교체.

### AI 서버 (Suh-Domain-AiServer)

`AiServerService.java`
- 모델 다운로드 재시도 — 2초 대기·최대 3회가 하드코딩
- IOException 시 상태를 문자열(`"completed"`)로 추측해 처리 → 불완전

**개선:** 재시도 간격·횟수를 설정값(ServerOptionKey)으로, 스트림 중단 처리 명확화.

### 도커 (Suh-Domain-Docker)

`DockerLogService.java`
- SSH 예외를 `e.getMessage().contains("timeout")` 같은 **문자열 매칭**으로 종류 판단(불안정)
- 실패 응답을 DTO의 `error` 필드로 반환 → 다른 곳은 예외 방식이라 불일치

**개선:** 예외 타입 기반 분기 + `CustomException` 일관 적용.

## 작업 우선순위 (이 문서 범위)

1. 소만사 버스 — 로그인/세션/예약 실패 명확화 (최근 이슈 직결)
2. 번역 — RuntimeException → CustomException (간단·효과 명확)
3. 깃허브 / 잔디 — 호출부 안전화 (잔디는 UI 대개편과 함께)
4. AI서버 / 도커 — 재시도·예외 분기 정리

> 각 항목은 개별 이슈로 등록 → 수정 → 검증 → 배포 순으로 진행 권장.
