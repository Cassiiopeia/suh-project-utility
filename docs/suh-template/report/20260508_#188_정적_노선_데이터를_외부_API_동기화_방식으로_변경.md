## 개요

`Suh-Domain-Somansa-Bus` 모듈의 노선 데이터를 부팅 시 하드코딩된 시드(`SomansaBusDataInitializer`) 대신 외부 buseezy(`csios.busin.co.kr/driving/drivelist.aspx`) HTML 응답을 Jsoup으로 파싱하여 동기화하는 방식으로 변경. `disptid` 기준 upsert + soft delete 로 FK 무결성을 보존하고, 동기화 트리거 회원은 `ServerOptionKey` enum + `ServerOption` entity 로 관리하여 admin 페이지에서 변경 가능하게 했음. 부수적으로 옛 도메인(`cs.android.busin.co.kr`) 을 신규 도메인(`csios.busin.co.kr`) 으로 일괄 갱신했고, 시스템 동적 설정값 관리 컨벤션을 `CLAUDE.md` 에 명문화함.

## 주요 구현 내용

### 외부 API 동기화 흐름

`drivelist.aspx` 가 HTML 페이지로 응답하므로 Jsoup 으로 파싱한다. 응답 구조:
- `.line-list li > a.cont` 의 `onclick` 속성: `goPage(disptid, '출근'/'퇴근', 'True'/'False')`
- `span.text` 의 텍스트: `06:55 당산역 1호 - 출근` 형식

정규식 2개로 파싱:
- `GO_PAGE_PATTERN`: `goPage\((\d+),\s*'([^']+)'\s*,\s*'(True|False)'\)`
- `LABEL_PATTERN`: `^\s*(\d{2}:\d{2})\s+(.+?)(?:\s+(\d+)호)?\s*-.*$`

### 충돌 처리: Upsert + Soft Delete

`SomansaBusSchedule` 과 `SomansaBusReservationHistory` 가 `SomansaBusRoute` 를 `@ManyToOne` FK 로 참조하므로 hard delete 불가. 따라서:
1. 외부 응답에 있는 `disptid` → DB 에 있으면 변경 감지 후 `update`, 없으면 `insert`
2. 외부에 없는 active 노선 → `isActive=false` (soft delete)
3. 외부에 다시 등장한 inactive 노선 → `isActive=true` (복구)

`syncTotalCount` 는 `routeRepository.count()` (active+inactive 전체 row).

### 트리거 회원 ServerOption 관리

동기화 호출 시 buseezy 로그인 세션이 필요하므로 어느 회원으로 로그인할지 결정해야 함. `ServerOptionKey.SOMANSA_BUS_SYNC_TRIGGER_LOGIN_ID` enum 항목으로 관리:
- 기본값(`defaultValue`): `chan4760@somansa.com`
- DB 의 `ServerOption` 테이블에서 변경 가능 → 기존 admin 페이지에서 자동 노출 (enum 자동 매핑)
- `getOptionValue()` 가 이미 Redis 캐싱(1시간 TTL) + default fallback 포함 → 재사용

향후 application.yml 하드코딩이 아닌 이런 시스템 동적 설정값을 일관되게 관리하도록 `CLAUDE.md` 에 컨벤션 섹션 신규 추가.

### 트랜잭션 분리

코드 리뷰 후 적용한 개선:
- `syncRoutes()` 자체는 non-transactional
- `fetchRemoteRoutes(triggerLoginId)` 가 외부 HTTP 호출 (login/createSession/fetchRouteList) 수행 — DB connection 점유 안 함
- `applyRemoteRoutes(remoteRoutes, triggerLoginId)` 만 `@Transactional` — DB upsert + soft delete 단계만 묶음

이전 구조에서는 `@Transactional syncRoutes()` 가 외부 HTTP 응답을 수 초~수십 초 기다리는 동안 DB connection 을 잡고 있어 connection pool 고갈 위험이 있었음.

### 셔틀 노선 처리

외부 응답의 셔틀 탭(`tab_sh`) 항목은 `goPage(_, _, 'True')`. DB 에 `isShuttle=true` 로 저장하되 예약 로직(`/reserve/manual`, 자동 예약 스케줄러) 에서는 별도로 필터링되어야 함 (현재는 본 PR 범위 외, 추후 #189 SSO 흐름 정상화 이슈에서 함께 다룸).

### Dashboard UI

기존 `somansaBusDashboard.html` 은 Thymeleaf 서버사이드 렌더링이었음. 동기화 후 즉시 갱신 표시를 위해 `loadRoutes()` / `displayRoutes()` JS 함수를 신규 추가하여 클라이언트 재렌더링 패턴 도입. 다른 함수(`loadStats`, `loadMembers`, `loadReservationHistory`) 와 동일한 `sendFormRequest(url, body, success, error)` 시그니처 따름.

## 코드 리뷰 후 수정 사항

| 항목 | 변경 |
|------|------|
| 트랜잭션 안에서 외부 HTTP 호출 | `syncRoutes` 비트랜잭션 + `applyRemoteRoutes` 만 `@Transactional` 분리 |
| `resolveTriggerLoginId` null 비교 3중 | `getOptionValue()` (default fallback + Redis 캐싱 내장) 사용으로 단순화 |
| `lastSyncedAt` active 노선만 → 부정확 | `findMaxUpdatedDate()` JPQL 쿼리 추가 (전체 row 대상) |
| `equalsNullable` 헬퍼 | `java.util.Objects.equals` 표준 메서드로 대체 후 헬퍼 제거 |

## 주의사항

### 첫 부팅 시 노선 0건
`SomansaBusDataInitializer` 제거 후 첫 부팅 시 노선 DB 비어있음. 관리자가 dashboard 에서 "노선 동기화" 한 번 클릭해야 채워짐. 부팅 자동 시드는 사용자 결정으로 도입하지 않음 (단일 진실 원천 = 외부 API).

### buseezy 잦은 로그인 차단 가능성
`syncRoutes()` 호출 시 매번 fresh login 수행 (기존 `executeReservation` 패턴 일관성). buseezy 가 잦은 로그인을 차단하면 별도 이슈에서 세션 캐싱 도입 검토 필요.

### 셔틀 노선 예약 분리 처리 미적용
`isShuttle=true` 노선이 자동/수동 예약 로직에서 제외되도록 하는 처리는 본 PR 범위 외. SSO 정상화 이슈 #189 또는 별도 후속 이슈에서 처리.

### admin 권한 검증
`/api/somansa-bus/route/sync` endpoint 가 별도 권한 어노테이션 없음. 운영 환경에서 admin only 정책 필요 시 후속 이슈에서 Spring Security 룰 추가 검토.

### 운영 시 외부 응답 비어있음 위험
`drivelist.aspx` 가 정상 응답이지만 `<ul></ul>` 비어있으면 → DB 의 모든 active 노선이 soft delete 됨. 운영 절차상 동기화 전 백업 또는 응답 노선 수가 임계값 이하일 때 경고하는 안전장치 도입 검토.
