## 📌 작업 개요

하드코딩된 서버 설정 값(청크 크기, AI 모델 등)을 DB 기반 동적 설정 시스템으로 전환. 사용자가 UI에서 설정을 수정하고 즉시 적용 가능하도록 구현.

**주요 구현 사항**:
- ServerOption 테이블 및 Enum 기반 설정 관리 시스템 구축
- 챗봇 청크 크기/중첩 크기 동적 설정 기능
- 전체 문서 재처리 기능 추가
- 설정 관리 UI 및 API 구현

---

## 🔍 문제 분석

### 기존 문제점
1. **하드코딩된 설정 값**: 청크 크기, 청크 중첩 등이 코드 내부에 상수로 고정
2. **테스트 불편**: 설정 변경 시 코드 수정 및 재배포 필요
3. **유연성 부족**: 다양한 AI 모델 테스트 시 코드 수정 불가피

### 해결 방향
- DB 기반 설정 관리로 런타임 설정 변경 가능
- Enum으로 설정 키 관리하여 타입 안정성 확보
- UI에서 설정 수정 및 즉시 적용 가능한 구조

## ✅ 구현 내용

### 1. Common 모듈 - ServerOption 인프라 구축

#### ServerOptionKey Enum 생성
- **파일**: `Suh-Common/src/main/java/me/suhsaechan/common/constant/ServerOptionKey.java`
- **내용**: 설정 키를 Enum으로 정의 (CHATBOT_CHUNK_SIZE, CHATBOT_CHUNK_OVERLAP)
- **특징**: 각 키마다 설명(description)과 기본값(defaultValue) 포함

```java
public enum ServerOptionKey {
  CHATBOT_CHUNK_SIZE("챗봇 청크 크기 (토큰 수)", "3000"),
  CHATBOT_CHUNK_OVERLAP("챗봇 청크 중첩 크기 (토큰 수)", "300");
}
```

#### ServerOption Entity 생성
- **파일**: `Suh-Common/src/main/java/me/suhsaechan/common/entity/ServerOption.java`
- **내용**: 설정을 저장하는 JPA 엔티티
- **특징**: 
  - `optionKey`에 Unique 제약 조건 설정
  - `BasePostgresEntity` 상속으로 생성일/수정일 자동 관리

#### ServerOptionRepository 생성
- **파일**: `Suh-Common/src/main/java/me/suhsaechan/common/repository/ServerOptionRepository.java`
- **내용**: JPA Repository 인터페이스
- **메서드**: 
  - `findByOptionKey()`: 설정 키로 조회
  - `existsByOptionKey()`: 설정 존재 여부 확인

#### ServerOptionService 생성
- **파일**: `Suh-Common/src/main/java/me/suhsaechan/common/service/ServerOptionService.java`
- **내용**: 설정 관리 비즈니스 로직
- **주요 메서드**:
  - `getOptionValue()`: 설정 값 조회 (없으면 기본값 반환)
  - `getOptionValueAsInt()`: Integer로 변환하여 반환
  - `setOptionValue()`: 설정 값 저장/업데이트
  - `initializeDefaultOptions()`: 앱 시작 시 기본 설정 자동 생성

#### DTO 클래스 생성
- **파일**: 
  - `Suh-Common/src/main/java/me/suhsaechan/common/dto/ServerOptionDto.java`
  - `Suh-Common/src/main/java/me/suhsaechan/common/dto/ServerOptionRequest.java`
- **내용**: API 요청/응답용 DTO

### 2. Chatbot 도메인 - 동적 설정 적용

## 🔧 주요 변경사항 상세

### 동적 설정 적용 흐름

1. **앱 시작**
   - `SuhProjectUtilityInitiation`에서 `initializeDefaultOptions()` 호출
   - DB에 기본 설정 자동 생성 (없는 경우만)

2. **문서 처리 시**
   - `DocumentService.processDocument()` 호출
   - `getChunkSize()`, `getChunkOverlap()`로 DB에서 현재 설정 조회
   - 조회된 설정 값으로 청크 분할

3. **설정 변경**
   - UI에서 "설정" 버튼 클릭 → 모달 열림
   - 현재 설정 값 로드 및 표시
   - 사용자가 값 수정 후 "저장"
   - `ServerOptionService.setOptionValue()` 호출하여 DB 업데이트

4. **전체 재처리**
   - "전체 재처리" 버튼 클릭
   - 모든 활성 문서를 새 설정으로 재처리
   - 결과 요약 표시

### 특이사항

**1. 에러 처리**
- 전체 재처리 시 개별 문서 실패해도 계속 진행
- 실패한 문서는 로그에 기록하고 결과에 포함
- 설정 값이 없으면 기본값 자동 반환

**2. 유효성 검사**
- 청크 크기: 500~10000 범위
- 청크 중첩: 0~1000 범위
- 클라이언트와 서버 양쪽에서 검증

**3. 트랜잭션 관리**
- 각 문서 재처리는 독립적인 트랜잭션
- 일부 실패해도 전체 롤백 방지

**4. 보안**
- `@PreAuthorize` 제거 (사용자 요청에 따라)
- 필요시 추후 추가 가능
