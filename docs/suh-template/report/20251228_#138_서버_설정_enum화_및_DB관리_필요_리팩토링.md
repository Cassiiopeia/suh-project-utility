### Phase 1: Common 모듈 - ServerOption 인프라

#### 1-1. ServerOptionKey Enum 생성
- **파일**: `Suh-Common/src/main/java/me/suhsaechan/common/constant/ServerOptionKey.java`
- **내용**: 설정 키를 Enum으로 정의
- **필드**: 
  - `CHATBOT_CHUNK_SIZE`: 청크 크기 (기본값 3000)
  - `CHATBOT_CHUNK_OVERLAP`: 청크 중첩 (기본값 300)

```java
public enum ServerOptionKey {
  CHATBOT_CHUNK_SIZE("챗봇 청크 크기 (토큰 수)", "3000"),
  CHATBOT_CHUNK_OVERLAP("챗봇 청크 중첩 크기 (토큰 수)", "300");
  
  private final String description;
  private final String defaultValue;
}
```

#### 1-2. ServerOption Entity
- **파일**: `Suh-Common/src/main/java/me/suhsaechan/common/entity/ServerOption.java`
- **특징**: 
  - `optionKey`에 Unique 제약
  - `BasePostgresEntity` 상속으로 생성일/수정일 자동 관리

#### 1-3. ServerOptionRepository
- **파일**: `Suh-Common/src/main/java/me/suhsaechan/common/repository/ServerOptionRepository.java`
- **메서드**: `findByOptionKey()`, `existsByOptionKey()`

#### 1-4. ServerOptionService
- **파일**: `Suh-Common/src/main/java/me/suhsaechan/common/service/ServerOptionService.java`
- **주요 메서드**:
  - `getOptionValue()`: 설정 조회 (없으면 기본값)
  - `getOptionValueAsInt()`: Integer 변환
  - `setOptionValue()`: 설정 저장/업데이트
  - `initializeDefaultOptions()`: 앱 시작 시 기본 설정 생성

#### 1-5. ServerOptionDto
- **파일**: `Suh-Common/src/main/java/me/suhsaechan/common/dto/ServerOptionDto.java`
- **특징**: **DTO 내부에 변환 로직 포함** (리팩토링 원칙)

```java
public static ServerOptionDto from(ServerOption option) {
    return ServerOptionDto.builder()
        .serverOptionId(option.getServerOptionId())
        .optionKey(option.getOptionKey())
        .optionValue(option.getOptionValue())
        .description(option.getOptionKey().getDescription())
        .build();
}

public static ServerOptionDto fromDefault(ServerOptionKey key) {
    return ServerOptionDto.builder()
        .optionKey(key)
        .optionValue(key.getDefaultValue())
        .description(key.getDescription())
        .build();
}
```

- **엔드포인트**:

  - `POST /api/server-option/list`: 모든 설정 목록
- **특징**: **Controller에 변환 로직 없음** (DTO의 `from()` 사용)

```java
public ResponseEntity<ServerOptionDto> getOption(@ModelAttribute ServerOptionRequest request) {
    ServerOption option = serverOptionService.getOption(request.getOptionKey());
    if (option == null) {
        return ResponseEntity.ok(ServerOptionDto.fromDefault(request.getOptionKey()));
    }
    return ResponseEntity.ok(ServerOptionDto.from(option));
}
```

### Phase 5: Controller 리팩토링 (CLAUDE.md 원칙)

#### 5-1. DTO 네이밍 변경 및 변환 로직 이동

**변경 전**:
- `DocumentDto` → 명확하지 않은 네이밍
- Controller에 `toDocumentDto()` 메서드 존재

**변경 후**:
- `ChatBotDocumentDto` → 명확한 네이밍
- DTO 내부에 `from()` static 메서드 추가

```java
// ChatBotDocumentDto.java
public static ChatBotDocumentDto from(ChatDocument document) {
    return ChatBotDocumentDto.builder()
        .documentId(document.getChatDocumentId())
        .title(document.getTitle())
        // ... 모든 필드 매핑
        .build();
}
```

#### 5-2. ChatBotDocumentController 분리

**변경 전**:
- `ChatbotController`에 챗봇 대화 + 문서 관리 API 혼재 (360줄)
- 5개 Response 타입 사용
- Controller에 변환 메서드 존재

**변경 후**:
- `ChatbotController`: 챗봇 대화 API만 (250줄)
- `ChatBotDocumentController`: 문서 관리 API만 (140줄, 신규)
- 각 Controller는 단일 Request/Response 사용
- 변환 로직 없음 (DTO의 `from()` 사용)

**ChatBotDocumentController 엔드포인트**:
- `POST /api/chatbot/document/list` - 문서 목록
- `POST /api/chatbot/document/detail` - 문서 상세
- `POST /api/chatbot/document/create` - 문서 생성
- `POST /api/chatbot/document/update` - 문서 수정
- `POST /api/chatbot/document/delete` - 문서 삭제
- `POST /api/chatbot/document/toggle-active` - 활성화 토글
- `POST /api/chatbot/document/reprocess` - 문서 재처리

#### 5-3. Request/Response 네이밍 통일

**변경 사항**:
- `DocumentRequest` → `ChatBotDocumentRequest`
- `DocumentResponse` → `ChatBotDocumentResponse`
- 명확한 네이밍으로 챗봇 문서 관리 DTO임을 표현
