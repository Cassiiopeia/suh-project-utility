## 🔴 1순위 — 지금 손봐야 할 것

### 1-1. 비밀번호·API 키가 코드에 그대로 적혀 있음 (보안)

`application.yml` 설정 파일에 **실제 비밀번호와 API 키가 그대로** 들어 있습니다.

- 서버 접속 비밀번호, 관리자 비밀번호
- 구글 AI / 내부 AI 서버 API 키
- 저장소 접속 정보

**왜 위험한가:** 이 파일이 외부에 노출되면 서버·계정·AI 비용이 그대로 털립니다. 가장 시급합니다.

**어떻게:** 이 값들을 코드에서 빼내 환경변수로 옮기고, 설정 파일은 버전 관리에서 제외합니다.

> ⚠️ 이건 단순 리팩토링이 아니라 **보안 조치**라, 키를 옮기는 동안 배포가 깨지지 않도록 신중히 진행해야 합니다. 별도로 다루는 걸 권장합니다.

### 1-2. 외부 연동 오류가 조용히 묻힘

소만사 버스·깃허브·번역 같은 **외부 서비스를 부를 때 오류가 나면, 로그만 남기고 그냥 "실패(-1, false, null)"로 넘어갑니다.**

- 소만사 로그인/예약, 깃허브 API 호출, 번역 호출 등
- 응답이 비어있을 때(NPE) 터질 수 있는 곳도 여러 곳

**왜 중요한가:** 이번 버스 예약 문제처럼 **"분명 켜놨는데 왜 안 되지?"**가 또 생깁니다. 오류 원인이 안 드러나니까요.

**어떻게:** 실패하면 명확한 예외를 던져서 화면·로그에 "무엇이 왜 실패했는지" 남도록 통일합니다.

## 🟠 2순위 — 곧 정리하면 좋은 것

### 2-1. 운영 중 못 바꾸는 고정값들

값이 코드에 박혀 있어서, 바꾸려면 매번 새로 빌드·배포해야 합니다. 운영 중 바로 바꿀 수 있게 설정값(관리자 페이지에서 편집)으로 빼야 합니다.

대표적인 것:
- 챗봇 신뢰도 기준값(0.7), 임베딩 모델명
- 도커 기본 컨테이너 이름, 스크립트 경로, 접속 타임아웃
- 캐시 유지 시간, 스케줄러 점검 주기
- 깃허브 잔디 자동 커밋 메시지·파일명

### 2-2. 똑같은 코드 복사붙여넣기

같은 로직이 여러 번 반복돼 있어, 한 곳을 고치면 나머지도 다 고쳐야 합니다 (빠뜨리면 버그).

- **AI 서버 모델 다운로드** — 거의 같은 코드가 430줄가량 두 벌 (가장 심함)
- **도커 로그 조회** — 비슷한 SSH 처리가 3~4번 반복
- 챗봇·잔디 쪽에도 중복 로직 일부

**어떻게:** 공통 부분을 하나로 묶고, 다른 부분만 갈아끼우는 방식으로 정리합니다.

### 2-3. 오래된 방식의 JSON 처리

깃허브 잔디·도커 쪽에서 JSON을 **문자열을 직접 잘라서** 읽는 곳이 있습니다 (코드에도 "원래는 제대로 된 방식 써야 함"이라고 적혀 있음). 형식이 조금만 바뀌어도 깨집니다.

**어떻게:** 표준 JSON 처리 방식으로 교체합니다.

## 🟡 3순위 — 여유 될 때

### 3-1. 오류 처리 방식 통일
- 곳곳에서 일반 `RuntimeException`을 던져, 사용자에게 500 에러로 나갑니다. 프로젝트 표준(CustomException)으로 통일하면 적절한 에러 화면/코드가 나갑니다.
- "이미 존재하는 사용자명", "번역할 내용 없음" 등은 400/409 같은 정확한 코드로.

### 3-2. 값 비교 방식 통일
- 여기저기 `== null` 직접 비교가 많습니다. 프로젝트 규칙(Optional, `Boolean.TRUE.equals()`)대로 통일.

### 3-3. 자잘한 정리
- 안 쓰는 코드·상수 제거 (예: 안 쓰이는 AI 모델 상수, 미사용 키 값)
- 화면(HTML)의 인라인 스타일 몇 군데 → CSS 클래스로
- 외부 통신 타임아웃 설정 추가 (응답 없을 때 무한 대기 방지)
