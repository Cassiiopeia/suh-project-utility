📝 현재 문제점
---

- i18n 기반 인프라(MessageSource, CookieLocaleResolver, 헤더 언어 토글, JS용 `window.MSG` 전역 주입)는 이미 구축·배포되어 정상 동작한다.
- 그러나 실제 다국어 적용은 header / footer / dashboard 일부에만 되어 있고, 나머지 16개 페이지의 텍스트는 여전히 한글로 하드코딩되어 있다.
- 영어로 전환해도 대부분의 페이지 본문은 한글 그대로 표시되어, 다국어 기능이 반쪽짜리 상태다.
- HTML 정적 텍스트뿐 아니라 JS 내부 한글 문자열(toast / alert / confirm / 동적 생성 텍스트)도 다수 존재해 함께 처리해야 한다.

🛠️ 해결 방안 / 제안 기능
---

- 나머지 전체 페이지의 사용자 노출 텍스트를 `messages_ko` / `messages_en` 기반 i18n으로 마이그레이션한다.
- HTML은 Thymeleaf 메시지 표현식(`th:text` / `th:placeholder` / `th:title`)으로, JS는 헤더에 주입된 `window.MSG` 전역 객체로 치환한다.
- 한국어/영어 두 언어를 모두 채운다.
- 자주 쓰는 핵심 페이지부터 그룹 단위로 단계적으로 진행한다.

⚙️ 작업 내용
---

- 페이지 그룹별 단계적 마이그레이션
  - 그룹 1(핵심): 번역기, SUH 랜덤, Task 트래커, 공지사항 관리
  - 그룹 2(관리): AI 서버 관리, 챗봇 문서 관리, Grass Planter, 세종대 인증
  - 그룹 3(소만사/기타): 버스 대시보드, 버스 회원 상세, DevOps 템플릿, 컨테이너 로그
  - 그룹 4(소량/마무리): 깃헙 이슈 도우미, AI 채팅, 대시보드 잔여, 헤더 잔여
- 각 페이지마다: 메시지 키 정의(한/영) → HTML 텍스트 치환 → JS 한글 치환 → 정합성 검증
- 번역 제외 대상(건드리지 않음): 고유명사(학력/자격증/회사/프로젝트명), 파일 경로, 식별자/이벤트명, 개발자용 console 로그, JS 값 비교에 쓰이는 문자열
